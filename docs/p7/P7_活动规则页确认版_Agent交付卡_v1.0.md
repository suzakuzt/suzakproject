## 二、页面目标

P7 的目标是把活动规则讲清楚，减少用户对「怎么获得次数、怎么点亮、怎么达成 985 礼盒资格、奖品是否必得」的疑问。

核心目标是：

1. 明确用户每日默认 1 次抽签机会；
2. 明确分享好友完成抽签后，会给原用户增加助力进度；
3. 明确每日分享最多获得 3 次额外机会；
4. 明确抽签会记录打卡 / 点亮进度，且每天最多点亮 1 天；
5. 明确累计点亮 7 天或邀请 5 位好友可解锁 985 和牛礼盒抽奖资格；
6. 明确奖品数量有限，以活动页面公示为准；
7. 明确活动是娱乐互动与祝福活动，不代表成绩预测；
8. 用企微二维码承接活动提醒、福利通知和私域沉淀。

## 五、状态逻辑

| 用户状态 / 页面状态 | 页面展示 | 点击动作 |
|---|---|---|
| 页面加载中 | 展示页面背景和规则卡 loading / 骨架 | 暂不影响返回按钮 |
| 页面加载成功 | 展示完整规则、企微二维码和说明文案 | 返回按钮正常；二维码可展示 / 预览 |
| 规则配置接口失败 | 展示本地兜底规则文案 | 记录异常埋点，不阻断用户阅读 |
| 规则配置缺失 | 使用前端兜底配置 | 不展示空白规则卡 |
| 二维码加载成功 | 展示企微二维码 | 用户可长按识别 / 截图识别 |
| 二维码加载失败 | 展示占位提示「二维码加载失败，请稍后再试」 | 可点击重试或等待刷新，具体按项目现有组件处理 |
| 用户点击返回 | 返回上一页 | 无历史栈时回 P1 首页 |
| 用户从 P6 进入 | 返回优先回 P6 | 以浏览器 / 小程序 webview 历史栈为准 |
| 用户直接打开 P7 | 返回回 P1 首页 | 防止空白或退出活动链路 |
| 页面内容超过一屏 | 支持纵向滚动 | 保持背景与安全区适配 |

## 七、埋点事件

| 事件名 | 触发时机 | 关键参数 |
|---|---|---|
| rules_page_view | 进入 P7 活动规则页 | activity_id、session_id、source_page |
| rules_page_render_success | P7 页面规则渲染成功 | activity_id、session_id、rules_count、has_qrcode |
| rules_page_render_fail | P7 规则配置加载失败或渲染失败 | activity_id、session_id、error_code、error_msg |
| rules_back_click | 点击左上角返回按钮 | activity_id、session_id、source_page、fallback_used |
| rules_qrcode_exposure | 企微二维码进入可视区域 | activity_id、session_id、qrcode_id |
| rules_qrcode_click | 点击二维码预览时触发，如果有预览能力 | activity_id、session_id、qrcode_id |
| rules_scroll_to_bottom | 用户滚动到页面底部 | activity_id、session_id |
| rules_qrcode_load_fail | 企微二维码加载失败 | activity_id、session_id、qrcode_id、error_code、error_msg |

## 八、异常处理

| 异常情况 | 页面处理 |
|---|---|
| 规则接口失败 | 使用前端兜底规则文案，同时记录 rules_page_render_fail |
| 规则列表为空 | 使用兜底 7 条规则，不展示空白卡片 |
| 二维码图片为空 | 展示「二维码暂未配置」或隐藏二维码图片区域，保留说明文案 |
| 二维码加载失败 | 展示加载失败占位，记录 rules_qrcode_load_fail |
| 返回无历史栈 | 跳转 P1 活动首页 |
| 页面样式在小屏溢出 | 支持纵向滚动，不压缩规则文字到不可读 |
| 文案过长 | 规则项自动换行，序号保持左侧对齐 |

3. 固定展示「活动规则」标题；
4. 固定展示「一举高中 · 六月牛气加油签」副标题；
5. 展示 1-7 条活动规则，规则文案按确认口径实现；
6. 第 4 条规则必须按最新口径写为「每日完成抽签后，自动点亮1天进度，每天最多点亮1天」，不要写成必须领取奖励后才点亮；
